## AI 모르면 간첩

요근래 가장 핫한 주제하면 바로 AI가 아닐까합니다. 핫하다고 말하기도 과소평가가 아닌가 싶은게, 산업과 경제, 그리고 모두의 꿈과 희망을 이끌고 있는 주체라고 볼 수 있을 것 같습니다.

굉장히 다양한 산업에 AI가 활용되며 많은 활용 사례들이 등장하고 있지만 이런 AI 활용의 최전선에는 아무래도 같은 계열인 개발자가 가장 영향을 많이 받고 있는데요. 저 역시도 실제로도 ~~이런 현상의 피해자로써~~ 여러 방면에서 적극적으로 사용해보고 있습니다.&#x20;

대표적으로 이 블로그도 [v0](https://v0.app/)라는 AI 툴을 뽑아서 초안을 뽑고 로직을 붙인 블로그이고, 블로그 포스팅을 작성할 때도 글의 전개에 대해 의견을 나누거나 작성된 글을 리뷰받기도 하며, 심지어 당장 이 글 역시도 AI와 함께 정리하며 작성하고 있습니다.&#x20;

특히나 최근에 모 대란이 발생하면서 여러가지를 시도해보고 있는데 이게 굉장히 재미있더군요. 내 아이디어가 요청 하나면 실제 제품으로 생산되는 것 자체가 생산자의 가치가 어디로 가냐와는 별개로, 생산 주체가 되기 쉬운 세상이 온 것은 틀림이 없습니다.

## 테스트는 통과하는데…

AI를 여러모로 사용해보면서 느낀 점은 만드는 과정을 획기적으로 단축시켜주지만, 이를 믿고 무작정 달리다보면 길을 잃는다는 점이었습니다. AI는 계속해서 코드를 뱉어냅니다. 나름의 논리를 바탕으로 적절한 코드를 작성하지만, AI가 한 번에 조망할 수 있는 양은 컨텍스트라는 한계가 존재합니다. 심지어 컨텍스트가 길어질수록 성능이 떨어지기까지 하니, AI에 의존적인 개발은 의도하든 아니든 어느 정도의 기도에 의존할 수 밖에 없습니다.

제가 이를 명확하게 깨달은 건 현재 개발하는 서비스를 AI만으로 한 번 개발해보려고 시도했을 때였습니다. 모든 테스트가 완벽하게 통과하는 코드가 동일한 스키마를 다른 방식으로 여러 군데서 중복 선언해서 임시로 땜빵해놓는 걸 보는 순간 AI는 더이상 든든한 동반자가 아니라 그저 코드를 뱉어내는 뽑기 기계가 될 뿐이었습니다.

더불어 사용자의 요청 사항을 끝까지 따르는게 아닌 중간에 타협해서 개발해버린다면… 프로젝트를 추적하는 일이 굉장히 힘들어지게 됩니다. 이런 현상이 특히 <Tooltip content="GPT 4.5 Fast 모드 주력 사용">Codex</Tooltip>에서 심한데, 모델 자체의 지능과는 별개로 Codex는 사용자의 요청이 클 경우 이를 안전하게 개발할 수 있도록 알아서 ~~멋대로~~ 작게 나누려는 경향이 있습니다.&#x20;

이 현상의 문제는 수많은 컨텍스트 위로 스쳐 지나간 말을 딸깍-하는 사용자가 일일이 기억하지 않는다는 점입니다. 때문에 실제로는 요청사항이 완료되지 않았는데 다음 개발을 진행하거나, 제대로 검증되지 않은 채로 개발이 진행되는 경우가 많았습니다.

이런 일이 겹치고 겹치다보면 내가 지금 어디까지 진행했고, 어떤 구조로 개발되어가고 있으며, 목표치에는 얼마나 다가갔는지 도무지 알 수가 없게 되는 사태가 발생합니다. 기능이 완성되기는 했는지도 의문이거니와, 결과물이라고 내놓은 내부 로직도 사실상 블랙박스가 되는거죠.&#x20;

이를 해결하기 위해 AI의 발전을 믿고 있을 수도 있지만, 안타깝게도 아직 그런 단계는 아닌 것 같습니다. 물론 근시일 내로 그런 시대가 올 것 같긴 합니다만 저는 현재를 살고 있으니까 해결 방법을 찾아야 했습니다.

## 기능은 개발 단위가 아니다

보통 우리는 AI에게 **행동 중심의 기능**을 개발해달라고 요청합니다. 예를 들어, **"프로필 업데이트 기능을 추가해줘"**&#xB77C;던가 아니면 **"메모 작성 기능을 추가해줘"**&#xC640; 같은 기능의 개발을 프롬프트로 요청하는거죠.&#x20;

문제는 이런 기능 단위가 건드리는 범위가 추적 가능한 범위와 일치하지 않는다는 것입니다. 프로필 업데이트 기능을 구현할 때도, 단순히 UI 하나에 끝나지 않고 데이터 베이스, 인증 로직, API 엔드 포인트까지 다양하게 뻗어나갑니다.

또한 각 기능 사이에는 선행 관계가 명확하지 않은 경우가 존재합니다. 쉽게 말해, A 기능이 개발되려면 B 기능이 개발되어야 하는데, B 기능의 일부 역시 A 기능이 개발되어야 개발할 수 있는 요상한 상태가 발생합니다.&#x20;

즉, 우리가 AI에게 요청하는 기능은 개발 과정을 추적하기에는 너무 크고, 순서대로 정렬할 수 없습니다.

## 숲에서 나침반 찾기

결국 문제는 AI에게 **무엇을** 개발해달라고 요청해놓고, 이를 **어떻게** 개발할지는 AI의 판단에 맡겨버렸다는 점입니다. 현재 AI라 불리는 LLM은 항상 동일한 판단을 내리지 않습니다. 즉, 동일한 상황에서도 내가 원하는 방향으로 개발을 할 수도, 안 할 수도 있습니다.

하지만 코드의 개발을 맡긴 이상 이런 비결정적 리스크는 안고 갈 수 밖에 없습니다. 결국 제가 생각한 해결 방법은 그보다 하위에서 계약을 고정하고, 추적 가능한 단위로 개발을 진행하여, 이해하기 쉬운 형태로 보고를 받는 것이 최선이라 생각했습니다.

제가 적용했던 구체적인 방법론은 다음과 같습니다.

1. **계약의 고정 (SSoT 확립)**

   AI가 참고할 수 있는 SSoT를 먼저 확립합니다. <Tooltip content="전체 어플리케이션의 기능을 먼저 주고 엔티티를 뽑으라고 하면 조금 더 쉽습니다.">엔티티</Tooltip>, <Tooltip content="TypeDoc을 활용해 작업이 끝날 때마다 추가된 API를 업데이트하도록 구성했습니다.">API 문서</Tooltip>, 아키텍쳐와 같은 문서를 md 파일로 작성하여 AI가 개발 중 바로바로 참고할 수 있도록 합니다. 물론 어려울 수 있습니다. 하지만 걱정 마세요. 이 역시도 AI와 대화하며 정립하면 됩니다.

   만들어진 모든 문서를 미리 참조할 경우 컨텍스트를 많이 차지할 수 있으므로, 적절한 폴더 구조를 만들어 두고 ssot-router 같은 [에어전트 스킬](https://developers.openai.com/codex/skills)을 개발해 적용해두면 편리합니다.
2. **원자적 개발 단위로 분해**

   기존 기능들을 세부 기능으로 구분하고 각 세부 기능별 필요한 엔티티와 함께 쭉 나열합니다. 이렇게 나열된 세부 기능을 **직접 작업/흡수/재평가/후속**으로 구분합니다. 이 역시  AI와 함께 작업하면 되고 반드시 정리된 결과를 문서로 남깁니다.

   **직접 작업**은 다른 작업에 흡수되지 않고 독자적으로 개발될 수 있는 기능이고, **흡수**는 다른 작업에 흡수되어 같이 개발되는 기능입니다. **재평가**는 개발이 진행되면서 선행 충족 사항이 만족했다면 개발에 들어갈 수 있는 작업입니다. 마지막으로 **후속**은 현재 목표하는 개발 목표에서는 개발할 필요가 없는 단위입니다.
3. **실제 개발 단위로의 정렬**

   이렇게까지 분류가 완료되었다면 이제 기능을 실제 개발 단위로 정렬할 수 있습니다. 각 세부 작업 중 직접 작업과 후속에 해당하는 작업들만 마일스톤 내에서 순서대로 개발할 수 있도록 분류 및 정렬합니다. 이것도 AI와 같이하면 됩니다. AI의 장점은 사람보다 많은 양을 한 번에 처리할 수 있다는거니까요. 그렇게 모든 작업들을 정렬했다면, 드디어 추적 가능하고 개발 가능한 개발 작업 단위가 완성되었습니다.&#x20;

   이제 이 마일스톤을 기준으로 개발을 시킵니다. 만약 마일스톤 내부의 작업이 모두 완료되었다면, 재평가 대상들을 훑어 현재 마일스톤의 범위에 포함되고, 선행 작업이 완료된 재평가 작업들에 한에 다시 평가를 진행해 직접 작업, 흡수로 변경합니다. 그리고 변경된 작업들은 작업 문서를 발행하고 해당 <Tooltip content="필요 시 마일스톤 내에서도 Wave로 개발 단위를 구분하면 조금 더 편합니다.">마일스톤 내에서 개발에 들어갑니다</Tooltip>. 더 이상 현재 마일스톤에서 재평가 가능한 작업이 없을 때까지 이를 반복합니다.
4. **추적 가능한 기록**

   앞서 말했듯 이 모든 과정은 반드시 문서로 남겨야 합니다. AI가 뱉어낸 코드를 그대로 수용하는 것이 아니라, 각 작업의 결과를 기록함으로써 프로젝트의 통제권을 유지하는 것입니다. 이때 문서는 다음과 같은 원칙을 따릅니다.

   * 가독성 우선: 서술형보다는 짧은 나열형으로 작성하여 AI와 사람이 핵심을 즉각 파악하게 합니다.
   * 미완료 작업 및 리스크 명시: 만약 기능을 완벽히 구현하지 못했다면, 남은 작업이나 잠재적 리스크를 반드시 기술합니다. 이는 다음 작업이나 마일스톤에서 AI가 참고할 중요한 컨텍스트가 됩니다.
   * 관점의 병행: 개발자의 기술적 관점뿐만 아니라, 실제 서비스가 어떻게 동작하는지 사용자 관점의 설명도 곁들입니다. ~~참고로 이 부분은 Claude가 Codex가 훨씬 잘합니다.~~

   AI 에이전트는 작업별 완료 문서, 마일스톤 별 문서, 그외 기타 SSoT 문서를 활용하여 개발 상황을 추적합니다. 이렇게 작성된 문서들은 컨텍스트와 독립적인 로컬 파일로 저장되어 필요 시 ssot-router 스킬을 사용해 에이전트가 참조할 수 있습니다. 더불어 저 같은 경우 md 파일로 작성하여 옵시디언을 통해 사람이 같이 읽을 수 있도록 구성하니 훨씬 편했습니다.

맞습니다. 자세히 뜯어보면 결국 이건 우리가 그동안 해왔던 개발과 다를 게 없습니다. 다만 더이상 사람만 공유하는 것이 아닌 AI와 함께 공유할 수 있는 자원이 된 것입니다. 이는 비결정적인 AI를 통제 가능한 체계로 편입시킨다는 큰 차이를 만듭니다.

이를 통해 AI는 정해진 규칙 위에서 자유롭게 사고하며, 정해진 문서를 통해 결과를 남기게 됩니다. 사람은 나열된 개발 단위를 보고 어디까지 개발되었는지 추적 가능하게 되고 프로젝트를 통제 가능하게 됩니다.

## 마무리

자, 이렇게 제가 AI와 개발하며 고민한 내용들을 풀어보았습니다. 기술은 눈부시게 발전하고 순식간에 코드를 뱉어내는 시대가 왔지만, 결국 개발자는 자신이 만드는 것을 이해하고 있어야 한다는 본질만큼은 변하지 않는 것 같습니다.

구조 없는 속도는 방향 잃은 질주에 불과합니다. AI가 아무리 똑똑해져도 그 지능이 뛰어놀 수 있는 정교한 궤도를 설계하는 것은 여전히 사람의 몫입니다. 물론 다른 쪽에선 이런 체계가 토큰을 낭비하는 원인이 된다거나 AI의 자율성을 저하한다는 의견도 보입니다만 옵시디언에 빼곡히 쌓인 마일스톤과 SSoT 문서들은 이제 저와 AI가 소통하는 공통의 언어가 되었습니다.

비록 '딸깍' 한 번에 모든 것이 해결되는 마법 같은 경험은 아니었지만, 내가 놓은 궤도 위를 달리는 AI를 보는 지금에야 비로소 저는 숨을 고를 수 있었습니다.

여러분은 지금 AI라는 거대한 숲에서, 본인만의 나침반을 들고 계신가요?
