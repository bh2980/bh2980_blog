\## URL 파라미터, 별거 아닌 것 같지만

쇼핑몰 프로젝트를 진행하면서 서비스와 어드민이 동일한 백엔드 API를 사용하는 구조를 선택했습니다. 두 프로젝트가 같은 API를 호출하는 만큼 요청 로직이 중복되는 것을 막기 위해, API 서비스 계층을 모노레포로 분리해 공유하도록 만들었죠. 덕분에 API 함수 하나를 고치면 두 프로젝트에 동시에 반영됐고, 중복 코드 없이 깔끔하게 유지할 수 있었습니다.

그런데 개발을 진행하다 보니 예상치 못한 곳에서 조금씩 불편함이 쌓이기 시작했습니다. 바로 \*\*URL 쿼리 파라미터\*\*였습니다. 페이지네이션, 필터, 정렬 같은 기능을 구현하다 보면 자연스럽게 URL 파라미터를 다루게 되는데, 생각보다 번거로운 일이 한두 가지가 아니었습니다.

\## 왜 이렇게까지 느슨한 거야

처음으로 신경 쓰이기 시작한 건 타입이었습니다. React Router의 \`useSearchParams\`가 반환하는 값은 \`string | null\`입니다.

\`\`\`tsx

const \[searchParams] = useSearchParams();

const page = searchParams.get('page') ?? '1';   // string | null

const limit = searchParams.get('limit') ?? '20'; // string | null

\`\`\`

파라미터를 꺼낼 때마다 null 처리를 직접 해줘야 했고, 숫자로 써야 하는 값도 문자열로 들어오니 변환도 따로 해줘야 했습니다. 파라미터가 많아질수록 이 보일러플레이트도 비례해서 늘어났고, 매 페이지마다 비슷한 코드를 반복해서 작성하는 게 굉장히 지루하게 느껴지더군요.

게다가 기본값 관리도 문제였습니다. \`page\`의 기본값이 \`'1'\`인지 \`1\`인지, \`limit\`의 기본값이 어디서는 \`20\`이고 어디서는 \`10\`인지 — 이런 값들이 각자 알아서 흩어져 있다 보니, 기본값을 하나 바꾸려면 코드베이스를 뒤져가며 여러 곳을 수정해야 했습니다.

\## nuqs를 꺼내들었지만

이 타입 문제는 저만 겪는 불편함이 아닌지, 이미 이를 해결해주는 라이브러리가 있었습니다. 바로 nuqs입니다. \`useState\`처럼 URL 파라미터를 다룰 수 있게 해주는 라이브러리로, 타입 변환과 기본값 설정을 한 번에 처리해줍니다.

\`\`\`tsx

const \[page, setPage] = useQueryState('page', parseAsInteger.withDefault(1));

const \[limit, setLimit] = useQueryState('limit', parseAsInteger.withDefault(20));

\`\`\`

딱 봐도 훨씬 깔끔합니다. 기대를 품고 프로젝트에 도입해봤는데… 이상한 동작이 발견됐습니다. nuqs와 React Router의 \`useSearchParams\`가 각자 별도의 객체를 바라보고 있는 건지, 같은 파라미터를 읽는데 두 훅이 반환하는 값이 다르게 나오는 것이었습니다. (v2 기준) 프로젝트 내 다른 곳에서 \`useSearchParams\`를 이미 사용하고 있었기 때문에, 두 훅이 동기화되지 않는 문제는 꽤 골치 아픈 일이었습니다.

물론 nuqs만으로 통일하거나, 다른 방법으로 이 동기화 문제를 해결할 수 있었을 겁니다. 그런데 막상 그 방향을 고민하다 보니 한 가지 사실을 깨닫게 됐습니다. 동기화 문제를 해결한다고 해서 진짜 문제가 사라지는 건 아니라는 것이었습니다.

\## 계약이 없다

서비스와 어드민은 같은 API를 쓰지만, URL 파라미터가 완전히 같지는 않았습니다. 예를 들어 \`currency\`는 서비스에서는 항상 특정 통화 코드가 필요했지만, 어드민에서는 전체를 조회하는 \`all\`이나 빈 값도 허용이 됐습니다. \`view\`는 두 프로젝트 모두 항상 포함되어야 하는 파라미터였는데, 기본값이 서비스에서는 \`main\`, 어드민에서는 \`admin\`으로 달랐습니다.

이런 파라미터들을 어디서 관리해야 할지가 막막했습니다.

공유 라이브러리인 API 계층에서 관리하자니, API 계층이 자신을 호출하는 프로젝트가 누구인지 알아야 했습니다. 서비스에서 호출했으면 \`view=main\`, 어드민에서 호출했으면 \`view=admin\`을 기본으로 넣어줘야 하니까요. 그러면 API 함수마다 "어느 프로젝트냐"에 따른 분기가 들어가게 됩니다. 공유 라이브러리가 단순한 HTTP 클라이언트 추상화가 아니라 프로젝트별 비즈니스 규칙을 담는 곳이 되어버리고, 새 프로젝트가 추가될 때마다 공유 코드를 손대야 하는 상황이 생깁니다.

그렇다고 각 프로젝트에서 알아서 처리하자니, \`currency\`와 \`view\`처럼 API 호출 시 항상 포함되어야 하는 파라미터들을 호출하는 모든 곳에 일일이 넣어줘야 했습니다. 하나라도 빠뜨리면 바로 버그가 됩니다. 또 \`page\`, \`limit\` 같은 공통 파라미터의 기본값을 바꾸게 되면, 두 프로젝트의 코드를 따로따로 찾아 수정해야 하고요.

어느 쪽도 마음에 들지 않았습니다. 문제의 본질은 타입이 아니었습니다. \*\*파라미터에 대한 계약이 어디에도 없다\*\*는 것이었습니다. 어떤 파라미터가 존재하는지, 타입과 기본값은 무엇인지, 프로젝트마다 어떻게 달라지는지 — 이것을 코드로 명시하고 강제하는 곳이 없었던 거죠.

\## 하나 만들어두고 조합하면 되지 않을까

문제를 이렇게 정리하고 나니, 해결 방향은 오히려 단순하게 보였습니다. 공통 파라미터는 한 곳에서 정의하고, 각 프로젝트는 거기에 자신의 파라미터를 얹는 구조면 됩니다. 기본값을 바꾸고 싶으면 그 정의를 찾아 바꾸면 되고, 프로젝트마다 다른 파라미터는 각자 추가하면 됩니다.

이 역할을 해줄 수 있는 라이브러리를 찾아보니 Zod가 가장 적합했습니다. 이미 프로젝트에서 API 응답 유효성 검사 등에 사용하고 있었고, \`extend\`와 \`merge\`를 사용하면 스키마를 쉽게 조합할 수 있었거든요.

\`\`\`tsx

// API 계층 (공유 라이브러리)

const BaseSchema = z.object(\{

&#x20; page: z.coerce.number().default(1),

&#x20; limit: z.coerce.number().default(20),

});

// 서비스 프로젝트

const ServiceProductSchema = BaseSchema.extend(\{

&#x20; currency: z.string(),                                        // 항상 필요, 기본값 없음

&#x20; view: z.string().default('main'),

&#x20; status: z.enum(\['active', 'inactive']).default('active'),

});

// 어드민 프로젝트

const AdminProductSchema = BaseSchema.extend(\{

&#x20; currency: z.string().default('all'),                         // 기본값 all 허용

&#x20; view: z.string().default('admin'),

&#x20; status: z.enum(\['active', 'inactive', 'deleted']).default('active'),

});

\`\`\`

\`BaseSchema\`는 API 계층이 소유합니다. 각 프로젝트는 이를 \`extend\`해서 자신만의 파라미터를 추가하거나 기본값을 덮어씁니다. \`currency\`처럼 기본값이 없는 필수 파라미터를 빠뜨리면 TypeScript가 컴파일 타임에 알려줍니다.

이 스키마를 실제 URL 파라미터와 연결하기 위해 \`useQuerySchema\`라는 커스텀 훅을 만들었습니다.

\`\`\`tsx

const \{ page, limit, currency, view, status } = useQuerySchema(ServiceProductSchema);

\`\`\`

스키마를 넘기면 URL에서 값을 꺼내 파싱하고, 없는 값은 스키마의 기본값으로 채워 반환합니다. 호출하는 쪽에서는 파라미터가 어떻게 파싱되는지, 기본값이 무엇인지 신경 쓸 필요가 없습니다. 그 모든 정보는 스키마 안에 있으니까요.

\## nuqs와는 다른 이야기

이쯤에서 드는 의문이 있을 수 있습니다. "결국 nuqs 써도 되는 거 아닌가요?" 하고요.

nuqs는 URL 파라미터를 React 상태처럼 편하게 다루는 문제를 풉니다. 타입 변환, 기본값, URL 동기화 — 이 레이어를 담당하는 라이브러리입니다. 반면 지금 만든 스키마 구조가 다루는 건 그 아래 레이어입니다. 어떤 파라미터가 존재하는지, 프로젝트 간에 어떻게 공유하고 확장하는지 — 이 \*\*계약\*\* 자체를 코드로 정의하는 것이죠.

둘은 경쟁하는 관계가 아닙니다. 실제로 지금이라면 \`useQuerySchema\` 내부에서 nuqs를 URL 동기화 레이어로 활용하는 구조도 충분히 가능합니다. 계약을 스키마로 정의하는 일과, 그 스키마를 URL에 연결하는 일은 별개의 문제니까요.

\## 마무리 — 파라미터 하나도 계약이다

돌이켜보면 처음의 불편함은 타입이 \`string | null\`이라는 지극히 표면적인 문제였습니다. 하지만 파고들다 보니 진짜 문제는 파라미터에 대한 계약이 어디에도 없다는 것이었습니다. 누가 소유하고, 어디서 정의하고, 어떻게 확장하는지 — 이 질문에 답이 없으면 파라미터 하나가 늘어날 때마다 혼돈이 조금씩 쌓입니다.

지금 \`useSearchParams\`로 파라미터를 다루면서 비슷한 불편함을 느끼고 있다면, 한 번쯤 물어보면 좋을 것 같습니다. 이 파라미터의 계약은 지금 어디에 있나요?
